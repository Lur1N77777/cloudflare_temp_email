# Initialize/Update D1 Database

When executing the wrangler login command for the first time, you will be prompted to log in. Follow the prompts to complete the login process.

## Initialize Database

```bash
cd worker
cp wrangler.toml.template wrangler.toml
# Create D1 and execute schema.sql
wrangler d1 create temp-email-db
wrangler d1 execute temp-email-db --file=../db/schema.sql --remote
```

> [!tip]
> Use a D1 database name such as `temp-email-db` or `cloudflare-temp-email-prod`.

After creation, you can see the D1 database in the Cloudflare console.

![D1](/readme_assets/d1.png)

## Update Database Schema

Before a `schema` update, confirm the currently deployed version and check the [Changelog](https://github.com/dreamhunter2333/cloudflare_temp_email/blob/main/CHANGELOG.md).

### 1. Back Up Remote D1 First

Export the database before every remote migration. Do not continue if the export fails:

```bash
cd worker
mkdir -p ../backups
wrangler d1 export temp-email-db --remote --output=../backups/temp-email-db-before-v1.10.0.sql
```

Replace `temp-email-db` with the database name used by your binding, then verify that the exported SQL file is non-empty and stored safely.

### 2. Run the Migration for Your Version

Find the `patch` file you need to execute and run it, for example:

```bash
cd worker
wrangler d1 execute temp-email-db --file=../db/2024-01-13-patch.sql --remote
wrangler d1 execute temp-email-db --file=../db/2024-04-03-patch.sql --remote
```

### v1.10.0 Migrations Dated 2026-07-13

The v1.10.0 authentication, concurrency, and IMAP changes use these eight migration files:

| Migration | Purpose | Repeatability |
| --- | --- | --- |
| `2026-07-13-address-token-version.sql` | Add `token_version` to address credentials | **Run once** (contains `ALTER TABLE address ADD COLUMN`) |
| `2026-07-13-auth-rate-limits.sql` | Create login-backoff state | Safe to rerun |
| `2026-07-13-inbound-delivery-index.sql` | Add inbound `delivery_key`, its unique deduplication index, and the address cursor index | **Run once** (contains `ALTER TABLE raw_mails ADD COLUMN`) |
| `2026-07-13-mail-flags.sql` | Create persistent IMAP system flags | Safe to rerun |
| `2026-07-13-outbound-send-requests.sql` | Create the outbound idempotency ledger | Safe to rerun |
| `2026-07-13-registration-challenges.sql` | Create one-time registration challenges | Safe to rerun |
| `2026-07-13-send-mail-quota-state.sql` | Create atomic daily/monthly send-quota state | Safe to rerun |
| `2026-07-13-user-token-version.sql` | Add `token_version` to user credentials | **Run once** (contains `ALTER TABLE users ADD COLUMN`) |

For a fresh upgrade where none of these eight files has run before, execute them in this order after the backup succeeds:

```bash
cd worker
wrangler d1 execute temp-email-db --file=../db/2026-07-13-address-token-version.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-auth-rate-limits.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-inbound-delivery-index.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-mail-flags.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-outbound-send-requests.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-registration-challenges.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-send-mail-quota-state.sql --remote
wrangler d1 execute temp-email-db --file=../db/2026-07-13-user-token-version.sql --remote
```

::: danger Do not rerun the three ALTER migrations
The complete `address-token-version`, `inbound-delivery-index`, and `user-token-version` files must each run only once; rerunning one fails because its column already exists. The other five migrations use `IF NOT EXISTS` or conflict guards and are safe to rerun. If migration history is uncertain, inspect the schema or restore the backup instead of treating a duplicate-column error as success.
:::
